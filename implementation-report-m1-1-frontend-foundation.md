## 1. The CSS Architecture

```
src/AiEng.Platform.App/Styles/
├── input.css       # Tailwind entry; the only file tailwindcss reads
├── tokens.css      # non-colour CSS custom properties (radius, shadow, font, type)
├── themes.css      # colour variables for light / dark / OS-preference fallback
├── base.css        # @layer base; body, type, focus, scrollbar, reduced-motion
└── utilities.css   # @layer utilities; .app-stack, .app-hstack
```

**`input.css`** is the only file the Tailwind CLI reads.
It orders the three `@import` directives **above** the
`@tailwind` directives (Tailwind v3 / `postcss-import`
drops `@import` nested inside `@layer`). The fourth
`@import` is inside `@layer utilities { … }` and
contributes the two project-specific utility classes.

**`tokens.css`** declares the design tokens that are
**not** colours on `:root`. The token names match the
Tailwind config's `theme.extend` block, so the utility
classes (`rounded-app`, `shadow-app-1`, `text-app-lg`,
`font-app-sans`, …) resolve to `var(--app-radius)`,
`var(--app-shadow-1)`, etc.

**`themes.css`** declares the colour variables. The
three blocks are:

1. `:root { … }` — light default.
2. `@media (prefers-color-scheme: dark) { :root:not([data-theme="light"]) { … } }` —
   OS preference, but only when the user has not made
   an explicit light choice. (The `:root:not([data-theme="light"])`
   selector intentionally does **not** exclude
   `[data-theme="dark"]` because the explicit override
   is already a stronger rule.)
3. `[data-theme="dark"] { … }` — explicit dark override.

The cascade order is: light is the base; OS preference
overrides light when no explicit attribute is set; the
explicit `data-theme` attribute wins over the OS
preference. The inline script in `App.razor` sets
`data-theme` **before** CSS is parsed, so the resolved
theme is in place at first paint.

**`base.css`** and **`utilities.css`** use `@apply`
exclusively, so the chrome markup reads in semantic
tokens (`bg-app-bg`, `text-app-fg-muted`, `rounded-app`)
without raw colour, font, or shadow values. `base.css`
honours `prefers-reduced-motion` and styles the
`::-webkit-scrollbar` with the surface tokens.

## 3. Theme Strategy

The theme is resolved **before CSS is parsed** by an
8-line inline script in `App.razor`. The resolution
order is:

1. `localStorage["app-theme"]` — an explicit `"light"`
   or `"dark"` value the user previously selected. This
   is the only value that wins over the OS preference
   once the user has expressed one.
2. `window.matchMedia("(prefers-color-scheme: dark)")`
   — the OS-level preference. Selected only if no
   stored value exists.
3. `"light"` — the default if both above are absent or
   fail (e.g. `localStorage` blocked, `matchMedia`
   unavailable).

The script writes the resolved value to
`<html data-theme="…">` and is the only JavaScript the
session introduces. It runs synchronously before
`<head>` finishes, so the theme is in place at first
paint and there is no flash of unthemed content.

The CSS rules in `themes.css` honour the cascade:

- `:root { … light variables … }` is the base.
- `@media (prefers-color-scheme: dark) { :root:not([data-theme="light"]) { … dark variables … } }`
  overrides the base when the user has **not** expressed
  an explicit light choice. The `:not([data-theme="light"])`
  guard means an explicit light value wins over the OS
  dark preference; an explicit dark value is already
  covered by the third rule below.
- `[data-theme="dark"] { … }` is the explicit dark
  override, which has higher specificity than the
  `:root` base and the `:root:not([data-theme="light"])`
  media query, so it always wins.
- `[data-theme="light"] { … }` is the explicit light
  override, included for symmetry and to anchor
  specificity in case the OS preference has resolved
  to dark.

The theme toggle button in `MainLayout.razor` is a
single `<button>` with an inline `onclick` that flips
`data-theme` and persists the new value to
`localStorage["app-theme"]`. It is the user-visible
expression of the M1.1 foundation; M1.2 (or the
`AppUserSettings` page in a later milestone) replaces
it with a proper component.

## 4. Build Approach

The build approach is the standard Tailwind v3 PostCSS
pipeline. The .NET build does not invoke `npm`; the
two pipelines are independent.

| Step | Command | Notes |
| ---- | ------- | ----- |
| Install | `npm install` | One-time, produces `package-lock.json`. |
| Production build | `npm run css:build` | `cross-env NODE_ENV=production` → `tailwindcss -i …/input.css -o …/app.css --minify`. PostCSS applies `tailwindcss`, `autoprefixer`, `cssnano`. |
| Development watch | `npm run css:watch` | Same pipeline without `--minify`, with `--watch`. cssnano is skipped because `NODE_ENV !== "production"`. |
| .NET build | `dotnet build` | Unchanged. The CSS path is read by Blazor from `wwwroot/css/app.css`; Blazor's Web SDK does not run the Tailwind build. |

The two-pipeline approach keeps `npm` (Node, network)
out of the .NET toolchain. CI / release engineering can
add an MSBuild `<Target Name="BuildCss">` later if a
one-command build is desired; M1.1 documents the
two-step build in the `wwwroot/css/README.md` and
leaves the choice open.

The `postcss.config.js` is intentionally minimal:

```js
module.exports = {
```

  `.slnx` file.
- **No CI / GitHub Actions / contributor-licence

## Deviations

- **Deviation 1 — CSS `@import` ordering fix.** The
  first version of `input.css` had the three
  `@import` directives inside `@layer base { … }`.
  The Tailwind v3 / `postcss-import` pipeline silently
  drops `@import` nested inside `@layer` directives;
  the resulting 5015-byte CSS contained none of the
  design tokens. The fix was to move the three
  `@import` directives to the **top** of the file,
  above `@tailwind base`. The post-fix build produces
  10,876 bytes with every design token present. The
  `wwwroot/css/README.md` documents the source-of-truth
  files; the bug and fix are not visible to consumers
  because the source files are the contract. Not a
  deviation from the brief; a self-correction during
  the session.
- **Deviation 2 — `NavMenu.razor.css` left in place.**
  The Blazor template's auto-generated scoped CSS file
  contains `.navbar-toggler`, `.nav-link`, and
  `.navbar-brand` rules. The brief says "Do not create
  component CSS files in this task" — it does not say
  "delete the template's component CSS." The updated
  `NavMenu.razor` markup no longer references any of
  these classes, so the rules are inert. The scoped
  CSS file is regenerated empty in M1.2 (or M2) when
  the side-nav is replaced by `AppSidebar`.
- **Deviation 3 — `@rendermode InteractiveServer` added
  to `Home.razor`.** The M1.0 template's `Home.razor`
  did not declare a render mode; `Counter.razor` did
  (`@rendermode InteractiveServer`). M1.1 added
  `@rendermode InteractiveServer` to `Home.razor` for
  consistency. (Strictly, this is not a class-attribute
  edit only; it is one Razor directive line. The
  change makes the page behave the same as the rest of
  the chrome and is the minimum required for
  consistency.)
- **Deviation 4 — `.editorconfig` added at the root.**
  Not explicitly required by the brief. Added to
  prevent the LF/CRLF line-ending flip on the first
  commit (the front-end tree uses LF; the .NET tree
  uses CRLF via `Directory.Build.props`-driven
  defaults). A two-line file; saves an editor
  configuration surprise.

## Known Limitations

- **No reusable components.** M1.1 builds the
  substrate; M1.2 builds the components on top of it.
- **Theme toggle is inline JS.** The button's
  `onclick` handler is a one-line inline script. M1.2
  introduces the first proper `App*` component (likely
  `AppButton`) and refactors the toggle to use it; a
  future session can move the handler into a small
  `theme.ts` module.
- **No `AppUserSettings` page.** The
  `localStorage["app-theme"]` value is the only
  persisted setting. A future milestone introduces the
  user-settings page that generalises the persistence
  pattern.
- **No web fonts.** The platform uses the OS system
  font stack. A custom typeface (if added later) is a
  one-line change to `--app-font-sans` and one
  `tailwind.config.js` line.
- **The pre-existing `prompts/` folder is unchanged.**
  As in M1.0, the repository contains a legacy
  `prompts/` folder that is not the canonical
  `.ai/prompts/`. M1.1 does not touch it.
- **The repository is not yet a git repository.** As
  in M1.0, `git status` returns "fatal: not a git
  repository." The `.gitignore` additions are in
  place; the first `git init` + initial commit will
  pick them up.
- **The `launchSettings.json` `http` profile binds to
  `localhost:5286`**, not the `5000` the plan
  mentioned. This is the default the .NET 10 Blazor
  template emits; M1.1 did not modify
  `launchSettings.json` (the plan's "no functional
  change" row).
