## 1. Scope and Out-of-Scope

### 1.1 In scope

- Build the **M1.2 reusable component library** —
  19 components across 5 categories
  (Primitives, Layout, Display, Feedback, Inputs).
  The full list is in § 5.
- Add bUnit test packages to the
  `AiEng.Platform.ComponentTests` project and
  write meaningful bUnit tests for the
  components that have non-trivial behaviour.
- Add a single architecture test
  (`Pages_Use_DesignSystem_Components_Not_DOM`)
  that is **narrow in scope** and only targets
  the three anti-patterns the brief and
  `ROADMAP.md` name.
- Ship the `/design-system` documentation page
  that consumes the new components and is the
  primary verification surface.
- Add a minimal navigation link to
  `/design-system` (the smallest possible entry,
  not a navigation registry).
- Update `docs/design-system.md` and
  `docs/component-guidelines.md` to reflect what
  shipped.

### 1.2 Out of scope (deferred to M1.3 or M2)

- **Final application shell** (sidebar, top bar,
  shell composition). M1.3.
- **Final navigation registry** (typed
  `INavRegistry`, route metadata, breadcrumbs).
  M2 per `ROADMAP.md`.
- **Full redesign of the five template pages**
  (Home, Counter, Weather, Error, NotFound).
  M1.2 leaves them on M1.1's chrome.
- **Breadcrumb system** (`AppBreadcrumb`).
  M2 per `ROADMAP.md`.
- **Form inputs beyond `AppInputLabel`** (e.g.
  `AppTextField`, `AppSelect`, `AppCheckbox`).
  M3 per `ROADMAP.md`.
- **The four composition-root architecture tests**
  (`Only_CompositionRoot_MayReference_ConcreteProviders`,
  `Pages_DoNotReference_ConcreteProviders`,
  `Application_DoesNotReference_ConcreteProviders`,
  `Components_DoNotInject_ConcreteProviders`).
  M4-D per `ROADMAP.md`.
- **Provider-specific components**
  (`AppProjectCard`, `AppProviderCard`,
  `AppModelPicker`, etc.). M3+.
- **M1.3 — Application Shell.** Explicitly not
  started.

The boundary is intentional: M1.2 is a
**library** slice, not a **page** slice. The
`/design-system` page is the only page the M1.2
session builds. The five template pages stay on
the M1.1 chrome; if a raw `<button>` exists in
them, M1.2's **narrow** architecture test is the
only place they are touched — and only to the
extent the test requires.

## 3. Selected Approach

The session follows `.ai/prompts/ui.md` and
`.ai/workflows/ui-design-review.md`. The M1.2
work is **foundation work**: build primitives
and containers first, then display, feedback,
and inputs, then the documentation page. The
order is non-negotiable per
`ui-design-review.md` § 2 Stage 3.

The session introduces **19 reusable components**
(18 from the brief + `AppErrorState` per
Correction 2), organised by the responsibility
folders defined in `docs/folder-structure.md`
§ 3.1:

- **Primitives** (`Components/Primitive/`):
  `AppButton`, `AppIconButton`, `AppBadge`,
  `AppStatusDot`, `AppDivider`, `AppStack`,
  `AppContainer`.
- **Layout** (`Components/Layout/`):
  `AppCard`, `AppSection`, `AppPanel`,
  `AppToolbar`.
- **Display** (`Components/Display/`):
  `AppPageHeader`, `AppAvatar`.
- **Feedback** (`Components/Feedback/`):
  `AppLoading`, `AppSkeleton`, `AppEmptyState`,
  `AppErrorState`, `AppAlert`.
- **Inputs** (`Components/Inputs/`):
  `AppInputLabel`.

The session adds bUnit to `ComponentTests`
(see § 6 for the verified package selection)
and writes meaningful tests for the components
that have non-trivial behaviour, are reused, or
have public-API surface that later milestones
will compose.

The session adds the `/design-system` route under
`Pages/DesignSystem.razor`. The page is the
**primary consumer** of the new components; it
demonstrates every variant, size, and state. A
single minimal nav link is added to
`NavMenu.razor` (a `NavLink` entry, not a
navigation registry — that is M2).

The session updates `docs/design-system.md` to
mark every implemented component with its
source-file reference (per ADR-015), bumps the
design-system version to 0.2.0, and updates
`docs/component-guidelines.md` to add patterns
for the four-state rule and the presentational
containers introduced.

The session does **not** redesign the M1.1
chrome. The five template pages stay on M1.1
markup. The single exception is the **narrow**
set of changes required to satisfy the new
architecture test — see § 7.

## 4. Alternatives Rejected

- **Alternative A — Build the application pages
  (Home, Counter, etc.) and extract components
  later.** Rejected. Per
  `ui-design-review.md` § 2 Stage 3, this is "the
  most expensive way to arrive at a design
  system." The session builds primitives and
  containers first; the documentation page is
  the only page the session builds.

- **Alternative B — Implement every M1 component
  in the catalogue (`AppLink`, `AppIcon`,
  `AppTooltip`, `AppDialog`, `AppDrawer`,
  `AppTabs`, `AppBreadcrumb`, `AppProjectCard`,
  `AppProviderCard`, etc.).** Rejected. The
  brief is explicit: "Required ... Primitives /
  Layout / Display / Feedback / Inputs" and
  "Do not add components not justified by
  current documentation." M1.2 implements the
  brief's list plus `AppErrorState` (Correction
  2), and **only** that list. `AppLink`,
  `AppIcon`, `AppTooltip` are M1 catalogue
  entries but the brief defers them; they are
  added when the page that needs them lands.
  M1.2's `AppEmptyState` and `AppAlert` cover
  the brief's "Empty states" and "Alerts"
  sections without needing an `AppIcon`.

- **Alternative C — Add the four composition-root
  architecture tests
  (`Only_CompositionRoot_MayReference_ConcreteProviders`
  et al.) in M1.2.** Rejected. Per
  `ROADMAP.md` M1 DoD, the four are
  "registered but disabled" with the activation
  point M4-D. No `Providers.<X>` project
  exists in M1.2, so the tests would have no
  real targets to fail. Adding them as active
  tests in M1.2 would be premature.

- **Alternative D — Make the M1.2 components
  data-owning with the four-state slots.**
  Rejected. Per `docs/design-system.md` § 5.4
  and ADR-014, the four-state rule is
  **conditional on data ownership**. The M1.2
  components are pure primitives and
  presentational containers; they render
  whatever the parent gives them. The
  four-state rule applies to the data-owning
  surfaces (the future list pages) and to the
  slots the data-owning parent exposes
  (`<Loading>`, `<Empty>`, `<Error>`,
  `<Populated>`). M1.2's `AppEmptyState`,
  `AppErrorState`, and `AppAlert` are the
  fallback surfaces the data-owning components
  compose; they are themselves primitives, not
  data-owning.

- **Alternative E — Redesign the M1.1 chrome
  (`MainLayout.razor`, `NavMenu.razor`) to
  compose the new components.** Rejected per
  Correction 1. M1.2 is a **library** slice,
  not a **page** slice. The chrome stays on
  M1.1 markup. M1.3 redesigns it.

- **Alternative F — Add semantic CSS classes to
  `Styles/base.css` or `Styles/utilities.css`
  for every component.** Rejected. Per
  `docs/component-guidelines.md` § 3.3, scoped
  CSS lives in `Component.razor.css` (the
  component's own scoped file). The shared
  `Styles/` files hold the design tokens, the
  theme variables, and the base layer;
  component-level `@apply` rules live in the
  component's scoped CSS file. This keeps the
  global CSS file small and the components
  self-contained.

- **Alternative G — Use bUnit 1.x packages
  (`bunit.core`, `bunit.web`,
  `bunit.runner.visualstudio`).** Rejected per
  Correction 3. bUnit 1.x is the legacy line
  (the 1.x `bunit.web` package and the
  `bunit.runner.visualstudio` package are
  superseded). bUnit 2.7.2 is the current
  stable and supports `net10.0` natively. See
  § 6.

### 6.4 Architecture test

A single new file:
`tests/AiEng.Platform.ArchitectureTests/Boundaries/PagesUseDesignSystemComponentsTests.cs`.

The test is a string-regex scan over files in
`src/AiEng.Platform.App/Pages/`. The test
asserts:

> No file under `src/AiEng.Platform.App/Pages/`
> contains a literal `<button>` (when
> `AppButton` or `AppIconButton` should be used),
> a literal `<input>` (when a design-system
> input equivalent exists), or inline
> `style="..."`.

The test rejects **only** the three anti-patterns
the brief and `ROADMAP.md` name. It does **not**
reject:

- headings (`<h1>`, `<h2>`, ...)
- paragraphs (`<p>`)
- tables (`<table>`, `<thead>`, `<tbody>`,
  `<tr>`, `<td>`, `<th>`)
- code blocks (`<code>`, `<pre>`)
- anchors (`<a>`)
- semantic sectioning elements (`<section>`,
  `<header>`, `<footer>`, `<nav>`, `<main>`,
  `<article>`, `<aside>`)
- form structure (`<form>`, `<label>`,
  `<fieldset>`, `<legend>`)

**Limitation explicitly documented:** M1.2 only
introduces `AppInputLabel` (a label), not
`AppTextField` (a labelled text field). The
architecture test therefore does **not** ban
literal `<input type="text">` elements in
M1.2's pages. The architecture test is the
**narrow** version described in Correction 4.
The full ban on literal `<input>` waits until
M3, when `AppTextField` (and its siblings) ship.

**Anti-pattern detection rules:**

| Anti-pattern | When to fail the build |
| ------------ | ---------------------- |
| Literal `<button` (case-insensitive) inside `src/AiEng.Platform.App/Pages/**` | When the `AppButton` or `AppIconButton` component is the correct replacement. The M1.2 architecture test fails on **any** literal `<button` in a page file (the M1.2 set is small enough that the M1.1 template's `<button>` in `Counter.razor` is the only one to be replaced). |
| Literal `<input` (case-insensitive) inside `src/AiEng.Platform.App/Pages/**` | M1.2 does **not** ban this. The M1.2 test asserts that no page file contains a literal `<button>`, but it does not assert on `<input>`. The full `<input>` ban is M3. |
| Inline `style="..."` (literal `style="` followed by anything but `>`) inside `src/AiEng.Platform.App/Pages/**` | Always fail. Inline `style` is forbidden by `STYLEGUIDE.md` § 5.4. |

If a future page genuinely needs a literal
`<button>` (none should, but the M1.2 test is
narrow), the test can be scoped to specific
files (a `// allowed` comment) or the test can
be moved to a "strict" mode in a later
milestone. M1.2 ships the narrow version.

## 13. Risks

- **Risk 1 — `bunit` 2.7.2 package restore
  failure on `net10.0`.** Likelihood: low.
  Impact: build failure, no fallback. Mitigation:
  the bUnit 2.7.2 package lists `net10.0` as a
  supported framework and resolves the
  `Microsoft.AspNetCore.Components` dependency
  to `10.0.0` on `net10.0`. If restore fails
  for any reason, the session stops, reports
  the failure, and asks the human. **No silent
  fallback.** Per Correction 3.

- **Risk 2 — Long utility chains in markup
  because the design tokens do not cover a
  needed combination.** Likelihood: low.
  Impact: review rejection. Mitigation: the
  M1.1 token set covers the M1.2 components'
  needs (verified by reading each component
  spec against the tokens). If a token is
  missing, the session adds it to
  `Styles/tokens.css` and `tailwind.config.js`
  together.

- **Risk 3 — The `AppContainer` component is
  ambiguous (Tailwind ships a `container`
  utility).** Likelihood: low. Impact: class
  collision in the compiled CSS. Mitigation:
  name the component `AppContainer` and the
  CSS class `.app-container`; the
  `@layer components` rule in Tailwind scopes
  the class.

- **Risk 4 — The architecture test has false
  positives on legitimate markup.** Likelihood:
  low. Impact: build failure. Mitigation: the
  test targets only the specific anti-patterns
  the brief names (literal `<button>`, inline
  `style="..."`). Tables, headings, anchors,
  code blocks, and sectioning elements are
  explicitly allowed. The test does **not**
  ban literal `<input>` in M1.2 per Correction
  4. See § 6.4.

- **Risk 5 — The M1.2 component count does
  not match the brief's "Required" list (plus
  `AppErrorState`).** The brief lists 18
  components. The plan lists 19 (18 + 1
  `AppErrorState` per Correction 2). If the
  implementation diverges, the count is
  re-verified at the end of the session.

- **Risk 6 — Scope creep into M1.3 territory.**
  Likelihood: medium (the temptation to
  "while we're here" restyle the chrome is
  real). Impact: deferred M1.3 work is
  prematurely done in M1.2. Mitigation: § 1.2
  is the explicit out-of-scope list; § 8.1
  is the explicit no-modify list; the
  implementation report's "Deviations"
  section flags any drift.

- **Risk 7 — `AppPageHeader`'s
  `Breadcrumbs` parameter is left as a
  parameter but unused, which can be
  mistaken for missing functionality.**
  Likelihood: low. Impact: code review
  confusion. Mitigation: the parameter is
  named `Breadcrumbs` (RenderFragment?),
  documented as M2-only, and renders nothing
  when null. The docstring lives in
  `docs/design-system.md` § 4
  (component catalogue).

## 14. Test Plan

The brief is explicit: "Do not chase coverage.
Write meaningful tests." The test plan covers
the components that have non-trivial behaviour,
are reused, or have public-API surface that
later milestones will compose.

### 14.1 bUnit tests

The 15 test files listed in § 6.3 cover the
following dimensions per
`docs/component-guidelines.md` § 6:

- Primary render.
- Each variant renders the matching class.
- Each size renders the matching class.
- Disabled state is applied when the parameter
  is true.
- Loading state is applied when the parameter
  is true.
- `ChildContent` is rendered when supplied.
- Custom slots are rendered when supplied
  (`AppCard` Header/Body/Footer; `AppSection`
  Title/Subtitle/Actions/Content; `AppPageHeader`
  Title/Description/Actions; `AppEmptyState`
  Title/Description/Icon/Actions;
  `AppErrorState` Title/Description/ErrorCode/
  CorrelationId/PrimaryAction/SecondaryAction;
  `AppAlert` Title/Description/Actions).
- Click event fires the `EventCallback` (for
  `AppButton` and `AppIconButton`).
- Accessibility assertions for `AppErrorState`
  (`role="alert"` present in rendered output)
  and `AppInputLabel` (`aria-required="true"`
  when `Required` is true).

Components that are pure presentational
containers with a single parameter (e.g.
`AppPanel`, `AppDivider`, `AppToolbar`) are
verified through the running app's visual
smoke test, not through bUnit. The brief's
"Do not chase coverage" rule applies.
