## 1. Objective

Populate the M2.1 `AppTopBarSlot`
with the **top bar**, the
**breadcrumb**, and the **page
header integration** the design
system already supports. The
session composes the M1.2
primitives (`AppToolbar`,
`AppPageHeader`, `AppStack`,
`AppButton`, `AppIconButton`,
`AppAvatar`, `AppBadge`,
`AppStatusDot`) and the M2.2
registry's `RouteMetadata.Parent`
chain to render the breadcrumb.

The session ships:

- The `AppTopBar` component (the
  real top bar that replaces
  M2.1's `AppTopBarSlot`).
- The `AppBreadcrumb` component
  (composed into `AppPageHeader`'s
  `Breadcrumbs` parameter, which
  was an M1.2 placeholder).
- The theme toggle moved into
  the top bar's `Trailing` slot
  (the M1.1 chrome had the theme
  toggle in the footer; M2.3
  relocates it).
- The user avatar in the top
  bar's `Trailing` slot.
- bUnit tests for the two new
  components.

## 2. Why It Is Separate

M2.3 is a distinct slice from
M2.1 and M2.2 because:

- The top bar is a **horizontal
  region** of the shell, not a
  vertical one. The layout
  composition (M2.1) does not
  care what the top bar
  contains; the top bar's
  content is its own concern.
- The breadcrumb depends on the
  M2.2 registry's `Parent` chain.
  Shipping the breadcrumb in
  M2.2 would couple the slice to
  a data source the slice does
  not own. Shipping it in M2.3
  gives the breadcrumb a stable
  data source.
- The page header integration
  is a cross-cutting concern
  (every page in M3+ uses it).
  Shipping it in M2.3 means
  every later slice composes
  the page header; the slice
  is the right home for the
  "every page has a page header"
  rule.
- The slice is small and
  independently deliverable:
  two new components, the
  theme toggle relocation,
  bUnit tests. The slice ends
  when the top bar renders,
  the breadcrumb follows the
  route's `Parent` chain, and
  the `AppPageHeader.Breadcrumbs`
  parameter is no longer a
  placeholder.

## 3. Dependencies

- **M2.1** (closed). M2.1
  ships the `AppTopBarSlot`
  placeholder M2.3 replaces.
- **M2.2** (closed). M2.2
  ships the registry the
  breadcrumb reads.
- **M0.5** (closed). The
  dashboard definition in
  `docs/dashboard.md` § 3.1
  names the top bar's slots
  (title, current time, theme
  toggle).
- **M1** (closed). The M1.2
  primitives `AppToolbar`,
  `AppPageHeader`, `AppStack`,
  `AppButton`, `AppIconButton`,
  `AppAvatar`, `AppBadge`,
  `AppStatusDot`.

## 4. Included Scope

- `AppTopBar` component
  (replaces `AppTopBarSlot` in
  `AppLayout`).
- `AppBreadcrumb` component
  (composed into
  `AppPageHeader.Breadcrumbs`).
- The theme toggle moved into
  the top bar's `Trailing`
  slot.
- The user avatar in the top
  bar's `Trailing` slot.
- bUnit tests for the two new
  components.
- The page header integration
  in the redesigned pages
  (the redesigns are M2.5;
  M2.3 ships the components
  the redesigns consume).

## 5. Excluded Scope

- **Sidebar items,
  `AppSidebarItem`,
  `AppNavItem`** → M2.2.
- **`INavigationRegistry` and
  `RouteMetadata`** → M2.2.
- **Self-awareness dashboard
  (live data)** → M2.4.
- **Final page redesigns** →
  M2.5.
- **Responsive matrix and full
  keyboard smoke test** → M2.5.
- **Layout root
  (`AppLayout`,
  `AppEmptyLayout`)** → M2.1.
- **Reconnect-modal changes.**
  The M1 template's
  `ReconnectModal.razor*` stays
  as-is. No M2.3 changes.
- **M3 / M4 work.** No
  `IProjectService`, no
  `IProjectStore`, no
  `Providers.<X>` projects.

## 6. Acceptance Criteria

- The top bar renders the
  title, the theme toggle, and
  the user avatar in the
  `Trailing` slot.
- The breadcrumb follows the
  current route's `Parent`
  chain (the chain read from
  `INavigationRegistry`).
- The theme toggle flips
  light / dark and persists
  the choice to
  `localStorage["app-theme"]`
  (the M1.1 chrome behaviour).
- The
  `AppPageHeader.Breadcrumbs`
  parameter is no longer a
  placeholder; every page
  that uses `AppPageHeader`
  gets a real breadcrumb.
- `dotnet build` → 0 warnings,
  0 errors.
- `dotnet test` → all bUnit
  tests pass; the new M2.3
  tests add to the count.
- `dotnet format
  --verify-no-changes` → clean.
- `npm run css:build` → clean;
  new top-bar / breadcrumb
  classes are in the compiled
  CSS.
- The app starts on
  `http://localhost:5286`; the
  top bar renders; the
  breadcrumb follows the
  current route's chain;
  light and dark themes
  render correctly.
- Keyboard smoke test: Tab
  through the top bar;
  every interactive element
  is reachable; focus is
  visible.
