## 1. Objective

Introduce the **navigation registry**
and the **registry-driven sidebar** in
the M2.1 `AppSidebarSlot`. The
registry is the typed data model the
sidebar, the breadcrumb, and the M3+
pages consume.

The session ships:

- The `INavigationRegistry`
  application service.
- The `RouteMetadata` data model
  record.
- The `[RouteMetadata]` attribute
  applied to every page in
  `src/AiEng.Platform.App/Components/Pages/`.
- The `RouteRegistry` implementation
  that scans the App assembly at
  startup.
- The composition-root extension
  method
  `NavigationServiceCollectionExtensions`
  in
  `src/AiEng.Platform.App/Composition/`.
- The `AppSidebar` component (the
  real sidebar that replaces
  M2.1's `AppSidebarSlot`).
- The `AppSidebarItem` component.
- The `AppNavItem` component.
- bUnit tests for the three new
  components.
- The `Pages_AreReachable_Through_Registry`
  architecture test
  (active in M2.2; was registered-
  but-deferred per `ROADMAP.md` § 4).

## 2. Why It Is Separate

M2.2 is a distinct slice from M2.1
because:

- The registry is a **data
  model**, not a **layout**. M2.1
  ships the layout shell; M2.2
  populates one of the regions
  (the sidebar) with the data
  the registry provides.
- The registry is consumed by
  every later milestone (M3, M4,
  M5, M6, M7, M8); a slice that
  introduces the registry without
  consuming the layout shell
  would be unmotivated.
- The
  `Pages_AreReachable_Through_Registry`
  architecture test asserts a
  property of the registry (every
  page in `Components/Pages/` has
  a `[RouteMetadata]` attribute
  whose `Href` matches a real
  route). The test belongs with
  the slice that introduces the
  registry, not with the layout
  slice.
- The slice is small and
  independently deliverable: a
  single assembly scan, a single
  attribute, three new
  components, one new
  architecture test. The slice
  ends when the sidebar reads
  from the registry and the
  test is green.

  (active in M2.2).

- **Top bar (`AppTopBar`)** →
  M2.3.
- **Breadcrumb
  (`AppBreadcrumb`)** → M2.3.
- **Page-header integration** →
  M2.3.
- **Self-awareness dashboard
  (live data)** → M2.4.
- **Final page redesigns** →
  M2.5.
- **Responsive matrix and full
  keyboard smoke test** → M2.5.

- **Reconnect-modal changes.**
  The M1 template's
  `ReconnectModal.razor*` stays

  `IProjectService`, no
  `IProjectStore`, no
  `Providers.<X>` projects.

## 6. Acceptance Criteria

- The registry is the data
  source for the sidebar; the
  sidebar reads from
  `INavigationRegistry` and
  renders one `AppNavItem` per
  route.
- Every page in
  `Components/Pages/` has a
  `[RouteMetadata]` attribute;
  the attribute's `Href`
  matches a real route in the
  project.
- The
  `Pages_AreReachable_Through_Registry`
  architecture test is
  **active and green**.
- `dotnet build` → 0 warnings,
  0 errors.
- `dotnet test` → all bUnit
  tests pass; the new M2.2
  tests add to the count; the
  new architecture test passes.
- `dotnet format
  --verify-no-changes` → clean.
- `npm run css:build` → clean;
  new sidebar component
  classes are in the compiled
  CSS.
- The app starts on
  `http://localhost:5286`; the
  sidebar renders the five M1
  template pages plus the
  `/design-system` page; the
  active state matches the
  current route.
- Light and dark themes render
  correctly.
- Keyboard smoke test: Tab
  through the sidebar; every
  sidebar item is reachable;
  focus is visible; the
  active item is visibly
  active.
