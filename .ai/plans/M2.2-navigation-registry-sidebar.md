## 0. Front Matter

**M2.2 — Navigation Registry and
Sidebar** is the second slice of
milestone M2 (Application Shell and
Navigation). It introduces the
**navigation registry** — a typed
data model that describes every
navigable route in the application —
and the **registry-driven sidebar**
that replaces M2.1's
`AppSidebarSlot` placeholder. The
registry is the data model the
sidebar (M2.2), the breadcrumb
(M2.3), and the M3+ pages compose
against. M2.2 is small and
independently deliverable: a single
assembly scan, a single attribute,
three new components, one new
architecture test, and the
application-side composition-root
wiring.

The slice ends when:

1. The sidebar reads from
   `INavigationRegistry` and
   renders one `AppNavItem` per
   route.
2. Every page in
   `Components/Pages/` has a
   `[RouteMetadata]` attribute
   whose `Href` matches a real
   route in the project.
3. The
   `Pages_AreReachable_Through_Registry`
   architecture test is **active
   and green**.

The M2.1 plan
(`.ai/plans/M2.1-application-shell-skeleton.md`)
is the substrate M2.2 composes
against. The M2.3 plan stub
(`.ai/plans/M2.3-topbar-breadcrumbs.md`)
consumes the registry's `Parent`
chain to build the breadcrumb.

**Why this matters.** Without a
registry, every surface that needs
to enumerate navigable routes — the
sidebar, the breadcrumb, the
self-awareness dashboard (M2.4), the
M3+ project pages — has to re-scan
the assembly or hard-code its own
list. The registry is the single
source of truth for "what is
navigable, what is its title, what
is its parent, what is its order".
The `Pages_AreReachable_Through_Registry`
architecture test makes that
contract machine-enforced: any page
without a `[RouteMetadata]` attribute
fails CI.

**Boundary with other M2 slices.**

- **M2.1** (closed) ships the shell

- **Sidebar (M2.2, this slice):**
  enumerate all routes sorted

- **Breadcrumb (M2.3):** follow
  the current route's `Parent`

  `AppBreadcrumb` with the
  current page as the last
  crumb and each parent as a
  preceding crumb.

  show the count of reachable

The M2.2 `INavigationRegistry` is
shaped to satisfy the M2.2 +
M2.3 use cases; M2.4 may extend
the interface or consume the
existing methods. The contract
is the minimum that M2.2 and
M2.3 need.

## 3. Selected Approach

The M2.2 approach is the
**assembly-scan + attribute**
pattern. A single attribute
(`[RouteMetadata]`) decorates
every page; a single
implementation (`RouteRegistry`)
scans the App assembly at startup
and produces a sorted, parent-aware
list. The composition root
registers `RouteRegistry` as a
singleton; the sidebar
(`AppSidebar`) injects
`INavigationRegistry` and renders
one `AppNavItem` per route.

The attribute carries all the
metadata the registry needs:

```csharp
[AttributeUsage(AttributeTargets.Class, AllowMultiple = false, Inherited = false)]
public sealed class RouteMetadataAttribute : Attribute
{
    public string Href { get; }
    public string Title { get; }
    public string? Description { get; }
    public string? Icon { get; }
    public int Order { get; }
    public string? Parent { get; }
    public string? BadgeText { get; }

    public RouteMetadataAttribute(
        string href, string title, int order = 0,
        string? description = null, string? icon = null,
        string? parent = null, string? badgeText = null) { ... }
}
```

The `RouteMetadata` data record
mirrors the attribute:

```csharp
public sealed record RouteMetadata(
    string Href, string Title, int Order,
    string? Description, string? Icon,
    string? Parent, string? BadgeText);
```

The `INavigationRegistry`
interface exposes a single read
method:

```csharp
public interface INavigationRegistry
{
    IReadOnlyList<RouteMetadata> Routes { get; }
    RouteMetadata? FindByHref(string href);
    IReadOnlyList<RouteMetadata> ChildrenOf(string? parentHref);
}
```

`Routes` is the flat list sorted
by `Order` then `Title`; the
sidebar iterates it. `FindByHref`
is for the breadcrumb (M2.3);
`ChildrenOf` is for the future
tree view (M2.4). The three
methods are sufficient for M2.2
and M2.3; M2.4 may extend the
interface.

The `RouteRegistry`
implementation uses
`AppDomain.CurrentDomain.GetAssemblies()`
(or a single
`typeof(RouteRegistry).Assembly`
plus a passed-in assembly list
in DI) to enumerate types with
`[RouteMetadata]`, instantiate
the attribute, and build the
list. The list is built **once**
at startup (singleton lifetime)
and is immutable thereafter.

The composition root:

```csharp
public static class NavigationServiceCollectionExtensions
{
    public static IServiceCollection AddNavigation(
        this IServiceCollection services, params Assembly[] scanAssemblies)
    {
        services.AddSingleton<INavigationRegistry>(
            _ => new RouteRegistry(scanAssemblies));
        return services;
    }
}
```

is called from
`AiEng.Platform.App/Composition/ServiceCollectionExtensions.cs`
with `Assembly.GetExecutingAssembly()`
so the App's pages are scanned.

The
`Pages_AreReachable_Through_Registry`
architecture test:

```csharp
public sealed class PagesAreReachableThroughRegistryTests
{
    [Fact]
    public void Pages_AreReachable_Through_Registry()
    {
        var pages = Types.InAssembly(typeof(RouteRegistry).Assembly)
            .That()
            .ResideInNamespace("AiEng.Platform.App.Components.Pages")
            .And()
            .Inherit(typeof(ComponentBase))
            .GetTypes();
        var registry = new RouteRegistry(typeof(RouteRegistry).Assembly);
        foreach (var page in pages)
        {
            var attr = page.GetCustomAttribute<RouteMetadataAttribute>();
            Assert.NotNull(attr);
            Assert.Contains(attr!.Href, page.Name.ToLowerInvariant()
                .Replace(".razor", ""));
        }
    }
}
```

The test asserts two things:
(a) every page in
`Components/Pages/` has a
`[RouteMetadata]` attribute, and
(b) the attribute's `Href` is
consistent with the file's
namespace and name. The second
assertion is a
sanity-check, not a hard
constraint (the test could be
relaxed later if needed).

## 4. Alternatives Rejected

### 4.1 Convention-based
configuration (no attribute)

Instead of `[RouteMetadata]`,
define the metadata in a single
`navigation.json` (or
`navigation.cs`) and have the
registry read it.

**Rejected because:**

- Decouples the metadata from
  the page; the two have to
  stay in sync by hand.
- The
  `Pages_AreReachable_Through_Registry`
  test cannot assert
  "every page has an entry"
  without a per-page marker.
- The attribute approach
  carries the metadata in the
  same file as the page
  markup, where a future
  contributor will look first.
- A `json` or `cs` file is
  a single point of edit; if
  it is missing or malformed,
  the whole sidebar is empty.
  The attribute approach is
  per-page; a missing
  attribute on one page
  yields a CI failure that
  names the page.

### 4.2 Source-generator
metadata

Use a C# source generator to
emit the `RouteMetadata` from
a single configuration file.

**Rejected because:**

- The source-generator
  pipeline is a non-trivial
  addition to the build
  (M0.5 already validated
  against the project
  graph; M2.2 should not
  expand it).
- The attribute approach is
  sufficient; the source
  generator would add
  indirection without solving
  a problem M2.2 has.
- Source generators are
  reviewed more carefully by
  the .NET compiler
  pipeline; an attribute is
  standard reflection.

### 4.3 Razor-directive
metadata

Add the metadata as a Razor
`@page`-adjacent directive
(`@routeMetadata { ... }`).

**Rejected because:**

- Razor directives are
  parsed by the Razor
  compiler; reading them
  via reflection requires a
  custom source generator or
  a Roslyn analyser.
- The C# attribute is
  visible to the same
  reflection the architecture
  test runs; the Razor
  directive is not.
- A `@routeMetadata` would
  be a new Razor syntax the
  project does not yet
  support; the C# attribute
  is universal.

### 4.4 Hard-code the
sidebar in `AppLayout`

Keep the M2.1 placeholder
structure; replace the slot
with a hard-coded list of
`AppNavItem`s.

**Rejected because:**

- The M3+ pages need a
  registry-driven
  enumeration; hard-coding
  in `AppLayout` defeats the
  purpose of the slice.
- A hard-coded list has to
  be edited every time a
  page is added; the
  architecture test cannot
  enforce the change.

### 4.5 `INavigationRegistry`
exposes `IEnumerable<RouteGroup>`

A nested group structure
(routes grouped by `Parent`)
instead of a flat list.

**Rejected because:**

- The M2.2 sidebar is a
  flat list; the
  breadcrumb is a chain
  (already handled by
  `Parent`). A `RouteGroup`
  type would be over-built
  for M2.2.
- M3+ may want groups; the
  interface can be extended
  later (the
  `IReadOnlyList<RouteMetadata>`
  property is a starting
  point). A migration to
  groups is a single PR
  when M3 introduces the
  first group.

## 13. Risks

- **Risk:** the assembly scan
  enumerates types from
  *every* loaded assembly, not
  just the App assembly. If
  the `AiEng.Platform.Application`
  assembly grows to contain
  pages (it currently does
  not), the scan would pick
  them up.

  **Mitigation:** `RouteRegistry`
  takes an `Assembly[]` list
  in the constructor; the
  composition root passes
  `new[] { typeof(Program).Assembly }`
  (the App assembly only).
  The Application project
  contains no `Components/Pages/`
  folder; the architecture
  test confirms this.

- **Risk:** the
  `[RouteMetadata]` attribute
  is not picked up by the
  test if the page inherits
  from a base class that
  also has the attribute. M2.2
  uses
  `AttributeUsage(Inherited = false)`;
  pages apply the attribute
  directly.

  **Mitigation:** the
  architecture test asserts
  the attribute is on the
  page type itself
  (`page.GetCustomAttribute<RouteMetadataAttribute>()`),
  not on a base class.

- **Risk:** the
  `Pages_AreReachable_Through_Registry`
  test uses `NetArchTest` (the
  M0.5 architecture-test
  library). The M0.5 baseline
  uses the
  `That().ResideInNamespace()`
  pattern; M2.2 follows it.

  **Mitigation:** the test
  file is in the
  `AiEng.Platform.ArchitectureTests`
  project; the M1.2 baseline
  is the reference.

- **Risk:** the new
  `Components/Navigation/`
  folder is not in the
  Tailwind content path; the
  new sidebar classes are
  purged out of `app.css`.

  **Mitigation:** the M2.2
  commit updates
  `tailwind.config.js` with
  the new content paths,
  mirroring the M2.1
  `Layouts/` addition.

- **Risk:** the
  `AddNavigation` extension
  is called with the wrong
  assembly (e.g. the
  Application assembly,
  which has no pages).

  **Mitigation:** the
  integration test asserts
  the App assembly is the
  one passed; the production
  call site is in
  `ServiceCollectionExtensions.cs`
  and passes
  `typeof(Program).Assembly`.

- **Risk:** the M2.2 sidebar
  is rendered inside
  `AppLayout`; if the
  `INavigationRegistry` is
  not registered, the
  sidebar throws at
  startup.

  **Mitigation:** the M2.2
  closeout commit fails
  `dotnet test` if the
  `ServiceCollectionExtensionsTests`
  integration test does
  not register the
  registry; the
  composition root calls
  `AddNavigation` before
  the App is built.

- **Risk:** the `ShowInSidebar`
  flag is a new property on
  the attribute and the
  record. Adding it now is
  a small change; adding it
  later (when M3 introduces
  the first hidden route)
  is a breaking change to
  the record.

  **Mitigation:** the M2.2
  plan adds `ShowInSidebar`
  to the attribute and the
  record in the same commit;
  the architecture test
  passes after the change.

## 14. Test Plan

### 14.1 Unit tests
(RouteRegistry)

- `Empty assembly returns empty list` —
  the
  `RouteRegistry` constructor
  takes an empty assembly
  list; `Routes` is empty.
- `Single attribute returns one entry` —
  an assembly with one
  `[RouteMetadata]` returns
  one `RouteMetadata`.
- `Multiple attributes return all entries sorted by Order` —
  an assembly with three
  attributes at `Order = 0`,
  `Order = 1`, `Order = 2`
  returns three entries in
  that order.
- `FindByHref returns the matching entry` —
  `FindByHref("/counter")`
  returns the
  `Counter.razor` entry.
- `FindByHref returns null for an unknown href` —
  `FindByHref("/nope")`
  returns null.
- `ChildrenOf returns the children of a given parent` —
  `ChildrenOf(null)` returns
  the top-level entries;
  `ChildrenOf("/admin")`
  returns the children of
  `/admin` (the M2.2 data
  has no parents, so
  `ChildrenOf("/admin")` is
  empty).
