## 1. Objective

Implement the **self-awareness
dashboard**: a read-only
`/dashboard` page the application
shows as its landing page. The
dashboard is data-owning for the
**structured-state-only** sections
it can already render in M2 (the
sections that consume
`.ai/state/*.json` directly, not
the sections that consume
provider- or service-derived
data).

The M2.4 dashboard is
**intentionally minimal**: the
sections that depend on M3+
(`IProjectService`, `IRunService`,
`IHistoryStore`,
`IProviderRegistry`, `IGitProvider`,
`IProviderHealthService`,
`IStreamingChannel`, etc.) render
their **Empty** state (per
ADR-014). The sections that depend
on the M0.5 structured state (the
**Current Milestone + Current Task**
section in
`docs/dashboard.md` § 3.2, the
**Build Status** section in § 3.9)
render their **Populated** state.

The session ships:

- A new page `/dashboard` at
  `src/AiEng.Platform.App/Components/Pages/Dashboard.razor`
  that uses the M2.1 `AppLayout`.
- A new application service
  `IProjectIntelligenceReader`
  in
  `src/AiEng.Platform.Application/ProjectIntelligence/`
  that exposes a typed
  `ProjectIntelligenceSnapshot`
  record (the current milestone,
  the current task, the current
  plan, the last commit, the
  branch, the build status, the
  test status, the M0.5
  structured-state summary).
- A composition-root extension
  method
  `ProjectIntelligenceServiceCollectionExtensions`
  in
  `src/AiEng.Platform.App/Composition/`
  that registers
  `IProjectIntelligenceReader`
  in the DI container.
- The data-owning sections of
  the dashboard that can be
  rendered from the M0.5
  structured state (per
  `docs/dashboard.md` § 3 — the
  sections that consume
  `.ai/state/*.json`).
- The **Empty** state for the
  sections that depend on M3+
  data sources; the **Empty**
  state names the slice that
  will fill the section
  ("M3 fills this section",
  "M4 fills this section",
  "M6 fills this section",
  etc.).
- bUnit tests for the
  dashboard's data-owning
  sections (each section
  renders the four slots per
  ADR-014).

## 2. Why It Is Separate

M2.4 is a distinct slice from
M2.1, M2.2, and M2.3 because:

- The dashboard is a **page**,
  not a shell region. The
  shell slices (M2.1, M2.2,
  M2.3) deliver the regions
  the page sits in; the page
  is its own slice.
- The dashboard's contract is
  in `docs/dashboard.md`
  (created in M0.5). The
  contract is the source of
  truth; M2.4 is the first
  implementation that
  satisfies the contract.
- The dashboard's read-only
  `IProjectIntelligenceReader`
  is the M2.4 abstraction the
  M3+ slices **do not modify**.
  M3+ slices add new data
  sources; they do not modify
  the reader. The boundary is
  what makes the slice
  independently deliverable
  (the M2.4 closeout's
  contract is final; the
  M3+ slices consume the
  contract).
- The slice is small and
  independently deliverable:
  one page, one service, one
  data record, one composition-
  root extension method, the
  Empty states for the M3+
  sections, bUnit tests. The
  slice ends when the
  dashboard renders the
  M0.5-data sections in the
  Populated state and the
  M3+-data sections in the
  Empty state.

- `IProjectIntelligenceReader`
  application service
  interface.
- `ProjectIntelligenceSnapshot`
  record.
- A `ProjectIntelligenceReader`
  implementation that reads
  `.ai/state/*.json` and
  produces the snapshot.
- The
  `ProjectIntelligenceServiceCollectionExtensions`
  composition-root
  extension method.
- The `/dashboard` page at
  `src/AiEng.Platform.App/Components/Pages/Dashboard.razor`.
- A new navigation item in
  the M2.2 registry for the
  `/dashboard` route.
- bUnit tests for the
  dashboard's data-owning
  sections (each section
  renders the four slots
  per ADR-014).
- The dashboard's
  **Empty** state for the
  M3+ data sources; the
  Empty state names the
  slice that will fill the
  section.

## 5. Excluded Scope

- **Layout root
  (`AppLayout`,
  `AppEmptyLayout`)** → M2.1.
- **Navigation registry
  implementation
  (`RouteRegistry`,
  `[RouteMetadata]`)** →
  M2.2. (M2.4 only **adds**
  a navigation item to the
  M2.2 registry; M2.4 does
  **not** introduce the
  registry.)
- **Top bar (`AppTopBar`)** →
  M2.3.
- **Breadcrumb
  (`AppBreadcrumb`)** → M2.3.
- **Final page redesigns of
  the five M1 template
  pages** → M2.5.
- **Responsive matrix and
  full keyboard smoke
  test** → M2.5.
- **M3 / M4 work.** No
  `IProjectService`, no
  `IProjectStore`, no
  `IProviderRegistry`. The
  M2.4 dashboard reads
  `.ai/state/*.json` only;
  it does not read
  `IProjectService`,
  `IHistoryStore`, etc.
- **Provider or runtime
  integration.** M2.4 does
  not introduce
  `IProviderHealthService`,
  `IStreamingChannel`, or
  any provider family.
- **Reconnect-modal
  changes.** The M1
  template's
  `ReconnectModal.razor*`
  stays as-is. No M2.4
  changes.

## 6. Acceptance Criteria

- The `/dashboard` route
  exists in the M2.2
  navigation registry.
- The
  `IProjectIntelligenceReader`
  service is registered in
  the composition root and
  resolves through DI.
- The dashboard renders the
  M0.5-data sections in
  the **Populated** state
  (per ADR-014): the
  current milestone, the
  current task, the
  current plan, the last
  commit, the branch, the
  build status, the test
  status.
- The dashboard renders
  the M3+-data sections
  in the **Empty** state
  with a clear "M3 fills
  this section" (or
  similar) message that
  links to the relevant
  M3+ plan or capability.
- Every data-owning
  section exposes the
  four slots per
  ADR-014.
- `dotnet build` → 0
  warnings, 0 errors.
- `dotnet test` → all
  bUnit tests pass; the
  new M2.4 tests add to
  the count; the
  architecture test
  `Application_DoesNotReference_ConcreteProviders`
  remains green (the
  dashboard resolves
  through services, not
  concrete providers).
- `dotnet format
  --verify-no-changes` →
  clean.
- `npm run css:build` →
  clean; the dashboard
  classes are in the
  compiled CSS.
- The app starts on
  `http://localhost:5286`;
  the `/dashboard` route
  returns 200; the
  dashboard renders the
  M0.5 data and the
  Empty states; light
  and dark themes render
  correctly.
- Keyboard smoke test:
  Tab through the
  dashboard; every
  interactive element
  is reachable; focus
  is visible.
