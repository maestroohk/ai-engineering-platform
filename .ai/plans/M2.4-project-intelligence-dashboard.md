## 1. Objective

Implement the **self-awareness
dashboard**: a read-only
`/dashboard` page the application
shows as its landing page. The
dashboard is data-owning for the
**structured-state-only** sections
it can already render in M2 (the
sections that consume
`.ai/state/*.json` directly, not
the sections that consume
provider- or service-derived
data).

The M2.4 dashboard is
**intentionally minimal**: the
sections that depend on M3+
(`IProjectService`, `IRunService`,
`IHistoryStore`,
`IProviderRegistry`, `IGitProvider`,
`IProviderHealthService`,
`IStreamingChannel`, etc.) render
their **Empty** state (per
ADR-014). The sections that depend
on the M0.5 structured state (the
**Current Milestone + Current Task**
section in
`docs/dashboard.md` § 3.2, the
**Build Status** section in § 3.9)
render their **Populated** state.

The session ships:

- A new page `/dashboard` at
  `src/AiEng.Platform.App/Components/Pages/Dashboard.razor`
  that uses the M2.1 `AppLayout`.
- A new application service
  `IProjectIntelligenceReader`
  in
  `src/AiEng.Platform.Application/ProjectIntelligence/`
  that exposes a typed
  `ProjectIntelligenceSnapshot`
  record (the current milestone,
  the current task, the current
  plan, the last commit, the
  branch, the build status, the
  test status, the M0.5
  structured-state summary).
- A composition-root extension
  method
  `ProjectIntelligenceServiceCollectionExtensions`
  in
  `src/AiEng.Platform.App/Composition/`
  that registers
  `IProjectIntelligenceReader`
  in the DI container.
- The data-owning sections of
  the dashboard that can be
  rendered from the M0.5
  structured state (per
  `docs/dashboard.md` § 3 — the
  sections that consume
  `.ai/state/*.json`).
- The **Empty** state for the
  sections that depend on M3+
  data sources; the **Empty**
  state names the slice that
  will fill the section
  ("M3 fills this section",
  "M4 fills this section",
  "M6 fills this section",
  etc.).
- bUnit tests for the
  dashboard's data-owning
  sections (each section
  renders the four slots per
  ADR-014).

## 2. Why It Is Separate

M2.4 is a distinct slice from
M2.1, M2.2, and M2.3 because:

- The dashboard is a **page**,
  not a shell region. The
  shell slices (M2.1, M2.2,
  M2.3) deliver the regions
  the page sits in; the page
  is its own slice.
- The dashboard's contract is
  in `docs/dashboard.md`
  (created in M0.5). The
  contract is the source of
  truth; M2.4 is the first
  implementation that
  satisfies the contract.
- The dashboard's read-only
  `IProjectIntelligenceReader`
  is the M2.4 abstraction the
  M3+ slices **do not modify**.
  M3+ slices add new data
  sources; they do not modify
  the reader. The boundary is
  what makes the slice
  independently deliverable
  (the M2.4 closeout's
  contract is final; the
  M3+ slices consume the
  contract).
- The slice is small and
  independently deliverable:
  one page, one service, one
  data record, one composition-
  root extension method, the
  Empty states for the M3+
  sections, bUnit tests. The
  slice ends when the
  dashboard renders the
  M0.5-data sections in the
  Populated state and the
  M3+-data sections in the
  Empty state.
