# M2.6 — M2 Closeout and Treehouse Dogfooding

> **The M2 closeout plan.** This plan promotes the M2.6
> summary entry (in the task board and in
> `milestones.json`) to a full closeout slice. The M2.6
> slice is the M1-style wrap-up for the M2 milestone:
> the verification, the gap-fixing, the deferred-review
> record, the project-continuity update, the handoff,
> the implementation report, **and the M2 retrospective
> (the first milestone retrospective in this
> repository)**.
>
> **Status:** Approved (2026-07-11; user pre-authorised
> in the M2.6 brief — the brief is the approval, no
> separate plan-review step).
>
> **Branch:** `feature/T-016-m2-closeout-and-treehouse-dogfooding`
> (created from `main` at the M2.5 closeout + infrastructure
> commit chain; per the new branching strategy in
> `.ai/workflows/branching-strategy.md`).

---

## 1. Why This Slice Exists

M2 has five implementation slices (M2.1 → M2.5) and
one closeout slice (M2.6). M2.6 is the M1-style wrap-up:
it verifies the M2 implementation against the ROADMAP
M2 definition of done, fixes the substantive gaps the
verification surfaces, records deferred reviews, and
introduces the M2 retrospective. The M2 retrospective is
**the first milestone retrospective in this repository**;
it lands in the same slice that introduces the
**Milestone Closeout Standard** (the canonical workflow
that requires every future milestone to ship a
retrospective).

M2.6 does not add new product functionality. M2.6 is
the **engineering hygiene** that closes the M2
milestone properly: a verified, evidence-backed, and
retrospected M2 that the next milestone (M3) can build
on with confidence.

## 2. In Scope

1. **Introduce the Milestone Closeout Standard** in the
   appropriate canonical AI workflow location. The standard
   is the single source of truth for "what a milestone
   closeout is" and the procedures every future milestone
   closeout must follow. The standard is documented once
   (per the anti-duplication rule in
   `.ai/workflows/branching-strategy.md`).
2. **Make milestone retrospectives a required deliverable
   for every future milestone closeout.** The Milestone
   Closeout Standard codifies this requirement; the
   standard's content is the only place the requirement
   is stated.
3. **Generate the M2 retrospective** with the 13 sections
   the user named in the brief (delivered capabilities,
   deferred capabilities, technical debt, known issues,
   lessons learned, architecture changes, documentation
   changes, testing summary, validation results,
   implementation reports, commit range, readiness for
   M3, recommendations for the next milestone). The M2
   retrospective lands in
   `retrospective-m2-application-shell-and-navigation.md`
   at the repository root.
4. **Validate the complete M2 milestone.** The validation
   gate is the same as the per-slice gate plus a
   milestone-level gate:
   - `npm run css:build` — the Tailwind + PostCSS
     pipeline emits `app.css` cleanly.
   - `dotnet restore` — every project's dependencies
     resolve.
   - `dotnet build` — 0 warnings, 0 errors.
   - `dotnet test` — every test passes (the registered
     axe-core and provider-boundary tests remain
     registered-but-disabled per ADR-016 / M4-D).
   - `dotnet format --verify-no-changes` — the format
     is clean.
   - **Visual smoke** — every M2 route returns 200 and
     the critical interactions (theme toggle click
     handler, sidebar navigation, breadcrumb walk,
     responsive matrix at the M2 primary viewport) work
     end-to-end.
5. **Update the project-continuity state per Rule 15**
   (the same per-slice state updates, plus the milestone-
   level updates): capability evidence, milestone
   evidence, task state, session state, structured JSON,
   markdown projections, ROADMAP.md, master delivery
   plan, current.md, task-board.md, latest handoff,
   milestone history, implementation report, retrospective.
6. **Coherent commit on the feature branch** with a
   focused, reviewable diff (per Rule 17).
7. **Fast-forward merge to `main`** per the branching
   strategy (rule 6: every completed task is merged
   into `main`; rule 7: the feature branch is deleted
   after the merge).
8. **Create the M2 milestone tag** (`m2`) at the
   M2 closeout commit on `main` (per rule 9 of the
   branching strategy: every milestone receives a
   milestone tag).
9. **Prepare the M3 plan** in `.ai/plans/` with
   `Status: Awaiting Approval`. Promote the first M3
   task to `Ready`.

## 3. Out of Scope

- **M3 implementation.** The brief explicitly says
  "do not expand into M3 implementation" and "Do not
  begin M3 implementation." M2.6 ends with the M3 plan
  in `Awaiting Approval` and the first M3 task in
  `Ready`; the M3 implementation lands in the next
  session.
- **Pushing to the remote.** The brief says "Push only
  if a remote is configured and pushing is authorised."
  The remote (`origin`) is configured but pushing is
  not authorised in this session; the prior session's
  rule (push requires explicit user authorisation per
  the command protocol) is preserved.
- **Renaming legacy feature branches.** The M0.5 → M2.5
  feature branches use the older
  `feature/<milestone>-<short-kebab>` naming. They
  remain as historical record per
  `.ai/workflows/branching-strategy.md` § 6.
- **Renaming the M2.6 branch to a legacy name** —
  the new branch follows the strategy's
  `feature/T-016-m2-closeout-and-treehouse-dogfooding`
  naming. After the merge, the branch is deleted (rule
  7).

## 10. Risks and Mitigations

- **Risk: the M2.6 plan is too long to execute in one
  session.** Mitigation: the work is mechanical
  (validation + state updates + docs); the 13-section
  retrospective is a single Markdown file. The session
  is bounded by the brief's acceptance criteria.
- **Risk: the new Milestone Closeout Standard
  conflicts with an existing rule.** Mitigation: the
  standard is reviewed against `AGENTS.md` (the
  constitution) and the existing workflows before
  commit; any conflict is resolved in favour of the
  constitution.
- **Risk: the M2 retrospective records technical debt
  that downstream milestones inherit.** Mitigation:
  the M3 plan accounts for the M2 retrospective's
  recommendations; technical debt is tracked, not
  buried.

The M2.6 closeout session stops after the M2 tag is
created on `main` and the M3 plan is in `Awaiting
Approval` with the first M3 task in `Ready`. The

`Awaiting Approval` → `Approve` → `In Progress`
