# M3.2 — Project Registration Slice 2 (Registration Form, Rename, Unregister)

> **The M3.2 plan.** M3.2 enables the actions
> that M3.1 wired to the seam: the **Register a
> project** form (a modal / inline form that
> creates a new project through
> `IProjectService.RegisterAsync`), the
> **Rename** action on the project card (a
> modal / inline form that updates the project
> name through `IProjectService.RenameAsync`),
> and the **Unregister** action on the project
> card (a confirmation step that removes the
> project through `IProjectService.UnregisterAsync`).
>
> **Status:** Awaiting Approval (2026-07-11;
> produced by the M3.1 closeout per the
> Progressive Coding Rule + the M3 plan § 2
> scope).
>
> **Branch:** (the M3.2 branch is created from
> `main` at the M3.1 closeout commit when M3.2
> starts; the branch is named
> `feature/T-019-m3-2-project-registration-slice-2`
> per the branching strategy rule 4).

---

## 1. Why This Slice Exists

M3.1 wired the M3 surface to the contract but
left the **Register**, **Rename**, and
**Unregister** actions **disabled** on the
seam. The actions are visible in the UI; the
buttons are rendered in the disabled state; the
underlying service methods are callable through
`IProjectService`. M3.2 enables the actions
end-to-end: a user can register a new project
(name + folder path), rename a registered
project, and unregister a registered project.

M3.2 is the **second M3 implementation slice**.
The M3 surface is the smallest piece of state
the platform needs to be useful on its own; M3.1
ships the read side; M3.2 ships the write side
of the project list.

M3.2 composes the M3.1 surface (`IProjectService`,
`IProjectList`, `AppProjectCard`); M3.2 does
**not** introduce new contracts or new
components in the M2 shell. M3.2 introduces the
**registration form** (a child of
`AppProjectList`'s Empty state + a modal
trigger on the page header's Register button),
the **rename form** (a modal triggered by the
card's Rename button), and the **unregister
confirmation** (a modal triggered by the card's
Unregister button).

## 2. In Scope

1. **Registration form.** A `RegisterProjectForm`
   component (or modal) that captures a **name**
   and a **path** (with a **Browse** button that
   uses the browser's file picker for the
   folder). The form calls
   `IProjectService.RegisterAsync(name, path,
   ct)`; on success the form closes and the
   list refreshes; on failure the form
   surfaces the `ValidationError` (one of
   `required`, `invalid_path`). The form is
   a child of the `/projects` page; the page
   header's **Register a project** button is
   the trigger.
2. **Rename form.** A `RenameProjectForm` modal
   that captures a **new name** (pre-filled with
   the current name). The form calls
   `IProjectService.RenameAsync(id, newName,
   ct)`; on success the form closes and the
   list refreshes; on failure the form
   surfaces the `ValidationError` (one of
   `required`, `not_found`). The modal is
   triggered by the `AppProjectCard.Rename`
   button (now enabled).
3. **Unregister confirmation.** A
   `ConfirmUnregisterProject` modal that asks
   "Are you sure you want to unregister
   *<name>*?" with **Cancel** + **Unregister**
   buttons. The modal calls
   `IProjectService.UnregisterAsync(id, ct)`;
   on success the modal closes and the list
   refreshes; on failure the modal surfaces
   the `ValidationError` (one of
   `not_found`). The modal is triggered by
   the `AppProjectCard.Unregister` button
   (now enabled).
4. **AppDialog composition (if missing).** If
   the M1.2 design system does not provide an
   `AppDialog` component, M3.2 introduces a
   minimal one: a backdrop + a content panel
   that traps focus, supports `Escape` to
   close, and respects `aria-modal` +
   `role="dialog"`. The `AppDialog` is a
   small composition: a `<dialog>` element
   (native HTML5 dialog) with the M1 design
   tokens applied. The M1.2 design system
   deferral is resolved.
5. **State management.** The
   `AppProjectList` is upgraded to support
   **mutations**: after a successful
   registration / rename / unregister, the
   list re-fetches (or, more efficiently,
   optimistically updates the in-memory
   snapshot). The list exposes a `Refresh`
   method; the form components invoke the
   refresh on success.
6. **Enable the actions on the card.** The
   `AppProjectCard.Open` button remains
   **disabled** (M4-A's responsibility per
   the M3 plan § 2.5). The
   `AppProjectCard.Rename` and
   `AppProjectCard.Unregister` buttons are
   **enabled** in M3.2 and wired to the new
   form / confirmation modals.
7. **Tests.**
   - Unit tests for the new validation
     paths (the M3.1 `IProjectServiceTests`
     already covers the `RegisterAsync` +
     `RenameAsync` + `UnregisterAsync`
     validation; M3.2 adds tests for the
     happy path with a `FakeClock`).
   - bUnit tests for
     `RegisterProjectForm` (every state:
     idle, submitting, success, validation
     error).
   - bUnit tests for
     `RenameProjectForm` (pre-fill,
     validation, success, not-found).
   - bUnit tests for
     `ConfirmUnregisterProject` (cancel,
     confirm, not-found).
   - bUnit tests for `AppProjectList`
     refresh on mutation.
   - bUnit tests for `AppProjectCard`
     buttons enabled + click handlers.
   - Architecture tests:
     `Forms_Resolve_Projects_Through_Service`
     (extends
     `Pages_Resolve_Projects_Through_Service`
     to cover the new form components).
8. **M3.2 implementation report + per-session
   handoff + state updates.** The M3.2 work
   follows the M1 / M2 / M3.1 closeout
   pattern: one coherent commit per slice;
   the implementation report at
   `implementation-report-m3-2-project-registration-slice-2.md`;
   the per-session handoff at
   `.ai/handoffs/2026-07-11-m3-2-project-registration-slice-2.md`;
   the state updates per Rule 15.
9. **M3 closeout (M3.x).** The M3 closeout
   slice produces the M3 retrospective at
   `retrospective-m3-project-registration.md`
   per the Milestone Closeout Standard
   (`.ai/workflows/milestone-closeout.md`).
   The M3 milestone tag (`m3`) is at the M3
   closeout commit on `main`. The M4-A plan
   is the M3 closeout's deliverable.

## 3. Out of Scope

- **M4 implementation.** M3.2 does not
  introduce `AiEng.Platform.Infrastructure`
  or any provider; the on-disk `IProjectStore`
  is M4-A's responsibility.
- **The Open action on the project card.**
  The Open action is **disabled** in M3.2
  (per the M3 plan § 2.5). M4-A enables the
  Open action (the durable store replaces
  the in-memory store, and the platform can
  resolve a process runner against the
  project's path).
- **The Open button is the only action
  that requires the durable store;**
  register, rename, and unregister all work
  against the in-memory store.
- **Provider implementation projects
  (`AiEng.Platform.Providers.<X>`).** M3.2
  does not introduce a single
  `Providers.<X>` project; M3.2 ships no
  provider. The first provider lands in M4-D.
- **Process execution.** M3.2 does not call
  `Process.Start`; M3.2 is a pure
  in-process state-management milestone. M4-A
  introduces `IProcessRunner` and the
  process boundary.
- **The axe-core audit activation.** The
  M2.5 `AxeCoreAuditTests` remain
  registered-but-disabled per ADR-016 /
  M4-D. The M3.2 plan does not activate the
  audit; M4-D is the activation milestone.
- **The provider-boundary tests
  activation.** Same — M4-D.

## 8. Risks and Mitigations

- **Risk: the M3.2 slice is too large to
  execute in one session.** Mitigation:
  the M3.2 work is bounded by the
  acceptance criteria; the form components
  are small and the tests are mechanical.
  M3.x (M3 closeout) is a separate slice
  per the Progressive Coding Rule.
- **Risk: the M3.2 forms bypass the M3.1
  `IProjectService` contract.** Mitigation:
  the `Forms_Resolve_Projects_Through_Service`
  architecture test enforces the
  single-seam rule; the form components
  inject `IProjectService` through
  constructor injection.
- **Risk: the M3.2 forms re-implement a
  component that the M1 design system
  already provides.** Mitigation: the M3
  acceptance criteria require the surface
  to compose the M1 design system; the
  `Pages_Use_DesignSystem_Components_Not_DOM`
  architecture test (M1) is in place; the
  `AppDialog` is the only new M1-style
  primitive, and only if M1.2 does not
  provide one.
- **Risk: the M3.2 forms introduce a new
  shell surface (a new layout, a new
  navigation pattern).** Mitigation: the
  M3.2 forms live in the existing M3
  surface; no new layout, no new
  navigation pattern. The forms are
  children of the existing M3.1
  components.
