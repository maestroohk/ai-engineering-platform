### 0.2 Objective

Introduce the **M2 layout root** as a
composition of the M1.2 design-system
primitives. The layout root defines the
shell's three regions — sidebar slot,
top-bar slot, content area — but does not
yet populate the sidebar or the top bar
with their M2.2 / M2.3 contents. The
five M1 template pages and the
`/design-system` page migrate from the
M1.1 chrome to the new layout root in
place; the migration is **only** the
layout wrapper change (no page content
redesign, no fake data, no M3 work).

### 0.3 Why This Task Is Next

- The application shell is the prerequisite
  for every later milestone. M3's
  project-registration page, M5's worktree
  list, M6's launch flow, M7's review
  panel, and M8's autonomous-loop and
  orchestration surfaces are all reached
  through the shell.
- The M1.1 chrome does not compose the
  M1.2 design system; it is the
  pre-design-system substrate. Without
  the M2 shell, the M1.2 components have
  no honest consumer.
- M2.1 is the smallest deliverable that
  makes the rest of M2 possible. M2.2
  (navigation registry and sidebar),
  M2.3 (top bar and breadcrumbs), and
  M2.4 (project-intelligence dashboard)
  all consume the shell regions M2.1
  ships.
- M2.1 is **chrome work**, not feature
  work. It does not introduce business
  capabilities; it introduces the
  substrate that the business capabilities
  compose.

## 3. Selected Approach

The session follows
`.ai/prompts/ui.md` and
`.ai/workflows/ui-design-review.md`.
The M2.1 work is **chrome work**:
build the layouts first, then the
placeholder slots, then the page
migrations, then the tests.

The M2.1 surface is **minimal by
intent**: the shell regions are
defined; the regions are populated
by M2.2 (sidebar items) and M2.3
(top bar, breadcrumb). M2.1 ships
the skeleton the next four slices
fill in.

The new M2.1 surface is:

- **`AppLayout.razor`** — the M2
  layout root. A three-region
  composition: sidebar slot
  (`AppSidebarSlot`), top-bar slot
  (`AppTopBarSlot`), and
  content area (the `@Body`).
  The composition uses
  `AppStack` and `AppContainer`
  for vertical rhythm and
  max-width wrapping. The
  composition is the design
  system's primary consumer.
- **`AppEmptyLayout.razor`** —
  the M2 alternate layout. A
  single-region composition:
  `AppPageHeader` and `@Body`.
  No sidebar, no top bar. Used
  by the `/design-system` page,
  `Error.razor`, and
  `NotFound.razor`.
- **`AppSidebarSlot.razor`** —
  the sidebar region of
  `AppLayout`. In M2.1 it
  renders an `AppEmptyState`
  with the M2.2 message. M2.2
  replaces the slot content with
  the registry-driven sidebar.
- **`AppTopBarSlot.razor`** —
  the top-bar region of
  `AppLayout`. In M2.1 it
  renders an `AppPageHeader` and
  an `AppAlert` with the M2.3
  message. M2.3 replaces the
  slot content with the M2.3
  top bar and breadcrumb.
- **`AppShellRegion.razor`** —
  a small presentational helper
  that renders the
  `data-app-region` attribute
  (e.g. `data-app-region="sidebar"`,
  `data-app-region="topbar"`,
  `data-app-region="content"`)
  and an `aria-label`. The
  helper keeps the M2 shell
  regions queryable for the
  M2.5 accessibility audit and
  for the M2.4 dashboard tests.

The M2.1 session does **not**
introduce the navigation
registry, the route metadata,
the sidebar items, the top bar,
the breadcrumb, the page
redesigns, the self-awareness
dashboard, or any architecture
test beyond the four already
in place. Those are the M2.2
through M2.5 slices.

  two-column grid.** Rejected.
  The M1.1 chrome uses raw
  Tailwind utility classes; it
  does not compose the M1.2
  design system. The M2 chrome
  is the design system's primary
  consumer; the M1.1 chrome is
  the pre-design-system
  substrate. Extending the M1.1
  chrome would be the most
  expensive way to arrive at
  the M2 surface.
- **Alternative B — Build M3
  (project registration)
  first.** Rejected. Per
  `ROADMAP.md` M2, the shell is
  the prerequisite for M3:
  every page in M3 is reached
  through the shell. Building
  M3 first would mean building
  the project-registration page
  with no shell, which would
  have to be re-skinned in M2
  anyway.
- **Alternative C — Bundle M2.1
  with M2.2 / M2.3 / M2.4.**
  Rejected. The M2
  delivery-reconciliation brief
  required non-overlapping
  slices. Bundling would
  re-introduce the overlap the
  brief explicitly removed.
  The slices are split so each
  is small, independently
  deliverable, and verifiable
  on its own.
- **Alternative D — Skip the
  layout root and ship a
  single-region shell.** Rejected.
  The M2 application shell is
  a three-region composition;
  skipping the layout root
  would mean re-introducing the
  composition in M2.2 / M2.3,
  doubling the work.
- **Alternative E — Wire
  `IReviewProvider` or any
  M7 surface.** Rejected. Per
  `ROADMAP.md`, M7 is the
  milestone that introduces
  the review families. M2.1
  does not introduce provider
  families; it introduces the
  application shell.

## 13. Risks

- **Risk 1 — The placeholder
  slots ship to production.**
  Likelihood: low. Impact: visual
  regression (the M2.1 ship shows
  "sidebar lands in M2.2" and
  "top bar lands in M2.3").
  Mitigation: the placeholders
  use `AppAlert` (Info variant)
  and `AppEmptyState` (a heading
  + a description). M2.2 and
  M2.3 remove the placeholders
  by replacing the slot
  contents.
- **Risk 2 — The M1.1 chrome is
  deleted before the new shell
  is ready.** Likelihood: low.
  Impact: the application does
  not start. Mitigation: the
  implementation order (§ 0.8)
  ships the new shell first,
  then deletes the M1.1 chrome
  atomically.
- **Risk 3 — The 77 M1.2 bUnit
  tests break because the
  M1.1 chrome's classes are
  gone.** Likelihood: low.
  Impact: regression. Mitigation:
  the M1.2 tests do not
  reference the M1.1 chrome's
  classes; the design-system
  tests render the components
  in isolation. Validation:
  `dotnet test` after the
  migration must show at
  least 81 passing tests.
- **Risk 4 — The M1 `Error.razor`
  and `NotFound.razor` pages
  behave differently in the new
  layouts.** Likelihood: low.
  Impact: the error and
  not-found pages render
  differently. Mitigation: the
  page content is unchanged; the
  M2.1 session verifies the
  visual smoke test for all
  five M1 routes.
- **Risk 5 — The
  `lavish-axi` M1 review is
  treated as a blocker.** Likelihood:
  low. Impact: review friction.
  Mitigation: M2.1 does not
  depend on the `lavish-axi`
  review; the review is a

  `.ai/state/task-board.md`. M2.1
  proceeds without it.
