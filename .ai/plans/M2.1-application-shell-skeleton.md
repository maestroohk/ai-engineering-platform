### 0.2 Objective

Replace the M1.1 chrome with the M2
application shell — `AppSidebar`, `AppTopBar`,
`AppBreadcrumb`, `AppNavItem`, `AppLayout`,
`AppEmptyLayout` — composed from the M1.2
design-system primitives. Introduce
`INavigationRegistry` and the route-metadata
data model. Redesign the five M1 template
pages to compose the new shell. Land the
`Pages_AreReachable_Through_Registry`
architecture test.

### 0.3 Why This Task Is Next

- The application shell is the prerequisite
  for every later milestone. M3's
  project-registration page, M5's worktree
  list, M6's launch flow, M7's review
  panel, and M8's autonomous-loop and
  orchestration surfaces are all reached
  through the shell.
- The M1.1 chrome does not compose the M1.2
  design system; it is the pre-design-system
  substrate. Without the M2 shell, the M1.2
  components have no honest consumer.
- The M2.1 plan is already authored
  (`Awaiting Approval`). The plan body in
  § 1 – § 17 was written by the M1 closeout
  session and reflects the M1.2 component
  catalogue and the ROADMAP M2 DoD. Approving
  the plan unblocks the first M2
  implementation session.

## 3. Selected Approach

The session follows `.ai/prompts/ui.md` and
`.ai/workflows/ui-design-review.md`. The M2.1
work is **chrome work**: build the shell
primitives first, then the registry, then the
breadcrumb, then the layouts, then the page
redesigns, then the documentation.

The session introduces the **M2.1 application
shell** as a composition of the M1.2 design
system. **No new M1.2 primitive is added**;
the shell is built by composing the existing
primitives and presentational containers
(`AppCard`, `AppSection`, `AppPanel`,
`AppToolbar`, `AppStack`, `AppButton`,
`AppIconButton`, `AppPageHeader`, `AppBadge`,
`AppStatusDot`, `AppAvatar`, `AppAlert`,
`AppEmptyState`, `AppErrorState`, `AppLoading`,
`AppSkeleton`).

The new M2.1 surface is:

- **`AppSidebar`** (`Components/Navigation/`) —
  the left rail. Composes `AppStack` (vertical)
  and `AppNavItem` (a new M2.1 primitive; see
  below). The sidebar reads its items from the
  `INavigationRegistry`.
- **`AppTopBar`** (`Components/Navigation/`) —
  the top bar. Composes `AppToolbar` (with
  `Leading` and `Trailing` slots) and the
  theme-toggle button (already shipped in M1.1
  chrome; M2.1 moves the toggle into the top
  bar's `Trailing` slot).
- **`AppBreadcrumb`** (`Components/Navigation/`)
  — the breadcrumb row. Composes
  `AppPageHeader`'s `Breadcrumbs` parameter
  (M1.2 placeholder) into a real component.
- **`AppLayout`** (`Layouts/`) — the M2 layout
  root. Composes `AppSidebar`, `AppTopBar`,
  `AppPageHeader`, and `@Body`. Replaces
  `MainLayout.razor`.
- **`AppEmptyLayout`** (`Layouts/`) — the M2
  alternate layout (for the design-system page
  and the error / not-found pages). Composes
  `AppPageHeader` and `@Body`; no sidebar, no
  top bar.
- **`AppNavItem`** (`Components/Navigation/`) —
  the sidebar item. Strongly typed `Href`,
  `Icon`, `Label`, `BadgeText`, `IsActive`.
  Renders an `<a>` (or `NavLink`) with the
  `AppButton` ghost-variant styling and the
  active state from `NavLink`'s `Match`
  parameter.

The **navigation registry** is the typed
source of truth for the sidebar:

- **`INavigationRegistry`** (`Application/`)
  — exposes `IReadOnlyList<RouteMetadata>`.
  The composition root
  (`App/Composition/`) registers the
  implementation (`RouteRegistry`). The
  registry is read by `AppSidebar` and by
  `AppBreadcrumb` (which derives its items
  from the current route's `Parent` chain).
- **`RouteMetadata`** (`Application/`) — a
  record: `Href`, `Title`, `Description`,
  `Icon`, `Order`, `Parent` (nullable, for
  the breadcrumb chain), `BadgeText`
  (nullable).
- **`[RouteMetadata]` attribute** —
  applied to every page in
  `Components/Pages/`. The composition root
  scans the assembly at startup and registers
  the metadata; the registry reads the
  attribute and the route from the page
  component.

The five M1 template pages are redesigned to
compose the new shell. `Counter.razor` and
`Home.razor` use `AppLayout`; `Error.razor`
and `NotFound.razor` use `AppEmptyLayout`.
`Weather.razor` uses `AppLayout` but the
fake-data table is replaced by an
`AppEmptyState` and an `AppAlert` (M2.1
ships the page with no real data; the
project-registration flow lands in M3).

The session updates
`docs/design-system.md` to mark the new
components as `Implemented (M2.1)`, the
`Source` column pointing to each component's
`.razor` file (per ADR-015). The
design-system version bumps from 0.2.0 to
0.3.0 (per ADR-015 § 10).

The session updates
`docs/component-guidelines.md` to document
the new M2.1 components, the layouts, the
navigation registry contract, and the
breadcrumb system.

## 4. Alternatives Rejected

- **Alternative A — Build the M2 chrome by
  extending the M1.1 two-column grid.** Rejected.
  The M1.1 chrome uses raw Tailwind utility
  classes; it does not compose the M1.2 design
  system. The M2 chrome is the design system's
  primary consumer; the M1.1 chrome is the
  pre-design-system substrate. Extending the
  M1.1 chrome would be the most expensive way to
  arrive at the M2 surface.
- **Alternative B — Build M3 (project
  registration) first.** Rejected. Per
  `ROADMAP.md` M2, the shell is the prerequisite
  for M3: every page in M3 is reached through
  the shell. Building M3 first would mean
  building the project-registration page with
  no shell, which would have to be re-skinned in
  M2 anyway.
- **Alternative C — Add a new M1.2 primitive
  (`AppIcon`) to the design system as part of
  M2.1.** Rejected. The M1.2 design system does
  not ship `AppIcon` (the M1.2 plan defers it
  to M1.3 / M2). M2.1's `AppNavItem` can use
  the existing typography tokens for its
  monogram-style icons (a single capital letter
  per route), avoiding the need for an `AppIcon`
  primitive. M2.1 records a follow-up task in
  `.ai/state/task-board.md` to add `AppIcon` in
  a later milestone.
- **Alternative D — Wire `IReviewProvider` or
  any M7 surface.** Rejected. Per `ROADMAP.md`,
  M7 is the milestone that introduces the
  review families. M2.1 does not introduce
  provider families; it introduces the
  application shell.
- **Alternative E — Skip the navigation
  registry and use a hard-coded list in
  `AppSidebar`.** Rejected. The
  `INavigationRegistry` is the data model M3
  consumes; introducing it in M2.1 is the
  pre-requisite for M3. Hard-coding the list
  would mean re-introducing the registry in
  M3, doubling the work.

## 13. Risks

- **Risk 1 — `INavigationRegistry` becomes a
  god-service.** Likelihood: low. Impact:
  design-system erosion. Mitigation: the
  registry is **read-only**; it is a data
  source, not an action surface. Mutations
  (e.g. adding a route at runtime) are out
  of scope; the registry is built at startup
  from the assembly's attribute scan.
- **Risk 2 — The M1.1 chrome's CSS leak.**
  Likelihood: low. Impact: visual regression.
  Mitigation: the M2.1 session deletes the
  M1.1 chrome's scoped CSS files; the M1.1
  theme tokens are retained, but the layout
  CSS is replaced.
- **Risk 3 — The `AppPageHeader.Breadcrumbs`
  parameter (M1.2 placeholder) becomes a
  dead parameter.** Likelihood: low.
  Mitigation: the M2.1 `AppBreadcrumb`
  component is composed into
  `AppPageHeader`'s `Breadcrumbs` parameter
  by `AppLayout`. The placeholder is no
  longer a placeholder.
- **Risk 4 — The redesign of the five
  template pages breaks the M1.2 architecture
  test.** Likelihood: low. Mitigation: the
  M1.2 `Pages_Use_DesignSystem_Components_Not_DOM`
  test asserts no literal `<button>` and no
  inline `style="..."`; the M2.1 pages do
  not introduce either.
- **Risk 5 — The five template pages' redesign
  introduces M3 work.** Likelihood: medium
  (the temptation to "while we're here" wire
  the project entity is real). Impact: M3
  work is prematurely done in M2.1.
  Mitigation: § 1.2 is the explicit
  out-of-scope list; § 8.1 is the explicit
  no-modify list; the implementation report's
  "Deviations" section flags any drift.
- **Risk 6 — The M1.1 `ReconnectModal` is
  deleted before an M2.1 replacement is
  ready.** Likelihood: low. Impact: the
  Blazor runtime's error UI is missing.
  Mitigation: the M2.1 `AppReconnectModal`
  is shipped in the same PR; the deletion is
  atomic with the replacement.
- **Risk 7 — The deferred `lavish-axi` review
  becomes a blocker for M2.1.** Likelihood:
  low. Impact: review friction. Mitigation:
  the M2.1 plan does not depend on the
  `lavish-axi` review; the review is a
  separate `Blocked` task in
  `.ai/state/task-board.md`. M2.1 proceeds
  without it.
